# 계산 플랫폼 선정: 로봇에 "계층형 두뇌" 장착하기

> ⚠️ 본 로드맵 내용은 공개 자료와 이론적 지식을 바탕으로 정리되었으며, 실기 검증을 거치지 않았습니다. 전기 및 기계 작업 시 안전 규정을 직접 확인하시기 바랍니다.

많은 사람들이 휴머노이드 로봇을 조립할 때 첫 번째 질문으로 "Jetson을 살까, NUC를 살까"를 묻습니다. 이것은 잘못된 첫 질문입니다. 올바른 첫 질문은 다음과 같습니다: 당신의 로봇이 실제로 어떤 종류의 계산 작업을 실행하며, 각각의 지연 한계와 실패 시 결과는 무엇인가? 선정의 본질은 어떤 보드의 TOPS가 더 높은지를 비교하는 것이 아니라, 작업을 계층화한 후 각 계층에 딱 맞는 연산 능력을 할당하는 것입니다. 너무 많으면 배터리와 예산을 낭비하고, 너무 적으면 가벼운 경우 보행이 불안정해지고 심하면 즉시 넘어질 수 있습니다. 이 페이지는 4단계로 구성됩니다: 계층화 → 플랫폼 선정 → 실시간성 방안 결정 → 엣지 VLA 추론 평가. 이론적 배경은 [제6장 계산, 전원 및 열 관리](/wiki/chapters/chapter-06/) 및 [제22장 소프트웨어 미들웨어](/wiki/chapters/chapter-22/)를 참조하십시오.

## 1단계: 연산 수요를 세 가지 계층으로 나누기

**【할 일】** 종이 한 장을 준비하여 계획 중인 모든 계산 작업을 나열하고, "제어 주기 × 실패 결과"에 따라 세 가지 계층으로 분류합니다:

- **L0 하위 실시간 계층**: 관절 서보 루프, 힘 제어, 안전 모니터링. 관절 전류 루프, 힘 제어 및 균형 제어는 일반적으로 0.5–2 ms의 하드 실시간 주기를 요구하며([제6장](/wiki/chapters/chapter-06/) 6.4.7절 참조), 시간 초과 시 불안정해집니다. 안전 관련 기능(비상 정지, 워치독, 과전류/과열 모니터링)은 메인 제어 장치와 독립된 [안전 MCU(Safety Microcontroller Unit)](/entry/ent_component_safety_mcu/)에 배치해야 합니다. 이는 메인 계산 장치가 다운되어도 동력을 차단할 수 있습니다.
- **L1 중간 계획 계층**: 상태 추정, 보행 생성, MPC(모델 예측 제어), 전신 제어(WBC). 일반적인 주파수는 50–500 Hz이며, 마이크로초에서 밀리초 수준의 지터는 허용되지만 안정적인 평균 처리량이 요구됩니다.
- **L2 상위 지능 계층**: 시각 인식, SLAM, 음성, VLA(시각-언어-동작) 모델 추론. 주파수는 1–30 Hz면 충분하며, 단일 프레임 지연이 조금 커도 "반응이 느릴 뿐" 직접적으로 넘어지지는 않습니다.

**【이유】** 세 계층의 제약 조건은 완전히 다릅니다: L0는 **결정성(determinism)** 이 필요하며, 연산 능력이 아닙니다; L2는 **피크 처리량(TOPS)** 과 메모리 대역폭이 필요하며, 실시간성 보장은 필요하지 않습니다. 실시간성을 보장하지 않는 x86 호스트에서 직접 1 kHz 힘 제어를 실행하는 것은 로봇의 균형을 운영 체제의 스케줄링 운에 맡기는 것과 같습니다. 반대로, 사용하지도 않을 TOPS에 비용을 지불하면 배터리 예산을 직접적으로 잡아먹습니다. ToddlerBot의 배터리 테스트에서는 기내 추론 시 "과열로 인한 클럭 다운까지" 단 19분만 지속되었습니다(출처: data/roadmap/research/toddlerbot.md). 계층화의 심층 원리는 이기종 컴퓨팅 트렌드입니다. 미래의 메인 컨트롤러는 CPU + GPU + NPU + 기능 안전 MCU의 이기종 조합이 될 것입니다([안전 MCU](/entry/ent_component_safety_mcu/) 카드의 트렌드 요약 참조).

**【당신의 상황 분석 방법】** 핵심은 액추에이터 방식에 달려 있습니다:

- **Dynamixel 버스 서보** 사용 시(위치 루프는 서보 펌웨어에 있음): L0는 이미 아웃소싱되었으며, 메인 컨트롤러는 수십 Hz의 위치 명령만 보내면 됩니다. ToddlerBot은 기성 통신 보드를 사용하여 2 Mbps 전송 속도에서 30대 모터의 전체 상태를 50 Hz로 피드백합니다(출처: data/roadmap/research/toddlerbot.md). 이 경우 L0 계층에는 특별한 하드웨어를 거의 구매할 필요가 없습니다.
- **자체 개발 준직구동(QDD) 액추에이터** 사용 시: 전류 루프/토크 루프는 드라이버 보드에 있지만, 상위 힘 제어 루프는 여전히 250 Hz–1 kHz가 필요합니다. Berkeley Humanoid Lite의 방식은 각 사지에 하나의 CAN 2.0 버스(1 Mbps)를 사용하고, 액추에이터와 IMU는 250 Hz로 통신합니다(출처: data/roadmap/research/berkeley-humanoid-lite.md). 메인 컨트롤러는 실시간 Linux를 실행하면 충분합니다.
- **풀사이즈 기종을 만들거나 기능 안전 인증이 필요한 경우**: 안전 MCU 계층은 생략할 수 없으며, ISO 13849, IEC 61508 등의 기능 안전 표준을 고려해야 합니다(출처: [안전 MCU](/entry/ent_component_safety_mcu/) 카드).

## 2단계: 후보 플랫폼 비교 및 실제 사례

**【할 일】** 1단계의 계층화 결과에 따라 L1+L2 계층(보통 통합 또는 분리)을 아래 네 가지 후보 중에서 주 컨트롤러로 선택합니다:

| 플랫폼 | AI 연산 성능 | 전력 소모 | 가격 | 생태계 | 실시간성 위치 |
|---|---|---|---|---|---|
| [NVIDIA Jetson AGX Orin](/entry/ent_component_nvidia_jetson_agx_orin_2024/) 64GB | 최대 275 TOPS (INT8) | 15–60 W 설정 가능 | 개발자 키트 약 1,999 USD (타사 참고 가격) | JetPack / Isaac ROS / Isaac Sim 통합 생태계, 16채널 MIPI CSI-2 | 소프트 실시간 (PREEMPT_RT 탑재), L1+L2 모두 처리 가능 |
| Jetson Orin NX 16GB | 최대 157 TOPS | 10–40 W | 공급업체에 직접 문의 필요 | AGX Orin과 동일한 생태계, 더 작은 크기 | 소프트 실시간, L2 위주, L1 겸용 |
| [NVIDIA Jetson Thor](/entry/ent_component_nvidia_jetson_thor/) | Blackwell 아키텍처, 엣지 VLA/VLM용 설계; 구체적인 TOPS는 공급업체에 직접 문의 필요 | 75–120 W급 (출처: [안전 MCU](/entry/ent_component_safety_mcu/) 카드) | 공급업체에 직접 문의 필요 | "물리적 AI"를 위한 플래그십 생태계 | L2 전용, L0/L1은 별도 구성 필요 |
| Intel N95 미니 PC | 전용 NPU 없음, AI 연산은 CPU/내장 그래픽 위주 (구체적인 TOPS는 공급업체에 직접 문의 필요) | 저전력 x86 (구체적인 TDP는 공급업체에 직접 문의 필요) | 약 129 USD (출처: data/roadmap/research/berkeley-humanoid-lite.md) | 표준 x86 Linux/ROS 생태계 | 소프트 실시간, L1 충분, L2는 소형 모델만 실행 가능 |
| Intel NUC (Core i3급) | 상동 | 상동 | 완제품 가격, 공급업체에 직접 문의 필요 | 성숙한 ROS/ROS2 생태계 | 소프트 실시간, 교육/개발용 |
| 라즈베리 파이 (Raspberry Pi 4) | AI 가속기 없음, CPU 전용 | 싱글보드 저전력 (구체적인 값은 공급업체에 직접 문의 필요) | 공급업체에 직접 문의 필요 | 커뮤니티 생태계 매우 큼, Python 친화적 | 소프트 실시간, L1 경량 제어 전용 |

(Orin 시리즈 연산 성능/전력 소모/가격 출처: [Jetson AGX Orin](/entry/ent_component_nvidia_jetson_agx_orin_2024/) 카드 및 경쟁 비교표.)

다음은 실제 오픈소스 프로젝트 5개가 어떻게 선택했는지 살펴봅니다 (출처는 모두 data/roadmap/research/ 조사 파일):

- **ToddlerBot (스탠포드) → Jetson Orin NX 16GB**: 기기에서 300M 파라미터 확산 정책 (약 100 ms 추론 지연 시간)과 10 Hz 스테레오 깊이 추정을 실행하려면 온보드 GPU가 필수이며, Orin NX는 성능/크기의 균형점입니다.
- **Berkeley Humanoid Lite → Intel N95 미니 PC (약 $129)**: RL 보행 정책의 네트워크가 매우 작아 N95로 하위 제어와 정책 배포를 동시에 실행하기에 충분하며, 예산 전액을 22개의 액추에이터에 투입했습니다.
- **Upkie → Raspberry Pi 4 + mjbots pi3hat (CAN 확장 보드)**: 바퀴-다리 균형 제어의 PID/MPC/RL 예제는 모두 소형 모델이며, 라즈베리 파이 + CAN 확장 보드가 가장 간편한 조합입니다.
- **ROBOTIS OP3 → Intel NUC (Core i3 듀얼 코어, 8GB DDR4, 250GB M.2 SSD) + OpenCR 서브 컨트롤러**: L1/L2는 x86 호스트, L0는 마이크로컨트롤러에 담당하는 전형적인 듀얼 보드 분업 구조입니다.
- **OpenLoong 청룡 → 400 TOPS 고연산 컨트롤러 + EtherCAT 버스**: 풀사이즈 43 자유도 공용 기기로, 상위 계층의 대형 모델 스케줄링에는 높은 연산 성능이 필요합니다 (2024 WAIC 발표 기준, 직접 페이지 확인 불가, 인용 시 주의).

**【이유】** 규칙은 명확합니다: **액추에이터가 "단순"할수록 (서보 모터), 정책이 작을수록 메인 컨트롤러는 저렴해집니다. 기기에서 대형 모델을 실행해야 Jetson이 등장합니다.** Orin 시리즈의 핵심 판매 포인트는 TOPS 숫자 자체가 아니라 "통합 생태계"입니다. JetPack SDK, Isaac ROS 및 Isaac Sim을 통해 시뮬레이션에서 훈련된 정책을 거의 원본 그대로 기기에 배포할 수 있습니다 (출처: [Jetson AGX Orin](/entry/ent_component_nvidia_jetson_agx_orin_2024/) 카드). 반면 [Jetson Thor](/entry/ent_component_nvidia_jetson_thor/)의 위치는 한 단계 더 나아갑니다: Blackwell 아키텍처는 기기에서 멀티모달 생성형 AI 및 VLA 정책을 실행하도록 특별히 설계되었지만, 75–120 W급 전력 소모는 단순한 "보드 업그레이드"가 아니라 방열, 배터리, 구조 전체를 재설계해야 하는 차세대 플랫폼임을 의미합니다.

**【당신의 상황 분석 방법】** 예산과 목표에 따라 세 가지 중 하나를 선택:

- **예산 < $500, 목표 RL 보행**: BHL 레시피 (N95급 미니 PC)를 그대로 따라 하여 예산을 액추에이터에 투자하세요. 단, 정책 네트워크가 작고 (MLP급) 시각 요구 사항이 낮아야 합니다.
- **예산 $1,000–2,000, 기기 내 추론/확산 정책/VLA 필요**: Orin NX 또는 AGX Orin. 먼저 비디오 메모리를 계산하세요: 모델 파라미터 수 × 양자화 정밀도 + 활성화 값. 16GB는 3억 파라미터급 확산 정책 (ToddlerBot 실증)을 수용할 수 있으며, 더 큰 VLA 모델은 64GB 버전이 필요합니다.
- **차세대 엣지 대형 모델 목표**: Thor를 주목하되, 방열 (75–120 W급은 이미 액체 냉각/상변화 논의 범위에 진입, 출처: [안전 MCU](/entry/ent_component_safety_mcu/) 카드) 및 배터리 용량을 구매 후 해결할 문제가 아닌 선결 설계 제약 조건으로 삼으세요.

## 3단계: 실시간성 방안 결정

**【할 일】** L0/L1 계층에 적합한 실시간 경로를 선택합니다. 주요 경로는 네 가지입니다:

1. **[Linux RT-PREEMPT](/entry/ent_software_rt_preempt_linux/)** : 메인라인 Linux에 실시간 패치를 적용하여 커널 코드의 대부분을 선점 가능하게 하고 인터럽트를 스레드화하여 수십 마이크로초 수준의 스케줄링 지연 시간을 제공합니다 (출처: [QNX](/entry/ent_software_qnx/) 카드 비교 절). 완전한 Linux 생태계를 유지하며 개인 및 연구소의 기본 선택입니다. 패치 적용 후에는 실제 부하 상태에서 `cyclictest`를 사용하여 최대 지연 시간을 측정해야 합니다 (방법은 [6장](/wiki/chapters/chapter-06/) 6.4.7절 참조).
2. **[EtherCAT](/entry/ent_technology_ethercat_2024/) 마스터 스테이션**: 실시간성을 버스로 내립니다. EtherCAT의 "플라잉 리드/라이트 (processing on the fly)"는 슬레이브가 프레임이 통과할 때 즉시 읽고 쓰도록 하며, 분산 클록 (Distributed Clocks)을 통해 모든 관절이 공통 시간 기준을 공유합니다 (출처: EtherCAT 카드). 메인 컨트롤러가 PREEMPT_RT + EtherCAT 마스터 스테이션 프로토콜 스택을 실행하면 1 ms급 관절 루프를 구동할 수 있으며, OpenLoong 청룡이 EtherCAT 버스를 채택했습니다 (출처: data/roadmap/research/openloong-qinglong.md).
3. **듀얼 커널/하드 RTOS**: Xenomai는 Linux 옆에 독립적인 실시간 커널을 실행하여 스케줄링 지연 시간을 마이크로초 수준까지 낮출 수 있지만 구성 및 유지 관리가 복잡합니다. [QNX](/entry/ent_software_qnx/)는 상용 마이크로커널 RTOS로, 파일 시스템, 네트워크 스택을 모두 사용자 공간 서비스로 실행하며 자동차 및 의료 분야에서 널리 사용되며 신뢰성과 인증 체계가 완벽하지만 라이선스 비용이 발생합니다 (출처: QNX 카드).
4. **MCU 하드 실시간**: 전류 루프, 안전 로직을 완전히 마이크로컨트롤러에 위임하고 (예: OP3의 OpenCR 서브 컨트롤러, 출처: data/roadmap/research/robotis-op3-darwin-op.md), 메인 컨트롤러는 소프트 실시간 작업만 수행합니다. 리소스가 제한된 노드는 Zephyr와 같은 오픈소스 RTOS를 사용할 수 있습니다 (출처: QNX 카드).

**【이유】** 표준 Linux의 커널 임계 구역은 우선순위가 높은 작업이 예측 불가능한 시간 동안 대기하게 하여 수십에서 수백 마이크로초의 지터가 1 ms 제어 루프에 치명적입니다 (원리는 [6장](/wiki/chapters/chapter-06/) 6.4.7절 참조). 실시간 방안 선택의 본질은 "결정성을 어느 계층에 둘 것인가"입니다: 커널 (PREEMPT_RT), 버스 (EtherCAT), 전용 OS (QNX), 또는 전용 칩 (MCU).

**【당신의 상황 분석 방법】** 결정 순서:

- 서보 모터 방식 → 어떤 실시간 패치도 적용하지 않고 로봇을 먼저 작동시키세요.
- 자체 개발 QDD + CAN 버스 (BHL 경로) → PREEMPT_RT로 충분하며, 비용이 저렴하고 자료가 많습니다.
- 자체 개발 QDD + 다중 관절 고동기화 요구 사항 (>20 관절, 1 ms 루프) → EtherCAT 마스터 스테이션, 초기 투자는 높지만 산업계에서 검증된 경로입니다.
- 향후 자동차/의료 등급 인증 목표 → 지금부터 QNX 개념을 알아두되, 개인 프로토타입 단계에서는 유료 도입이 필요하지 않습니다.

## 4단계: 엣지 VLA 추론 요구사항 평가

**【무엇을】** 로드맵에 "로봇이 명령을 이해하고 스스로 작업을 수행한다"는 내용이 있다면, [엣지 VLA 추론(On-Device VLA Inference)](/entry/ent_tech_on_device_vla_inference/)을 위한 전용 연산 능력 예산을 수립해야 합니다: 목표 VLA 모델의 파라미터 수, 양자화 방식(INT8/FP16), 요구되는 동작 출력 빈도를 나열하고, 이를 바탕으로 VRAM 및 TOPS 요구사항을 역산합니다. VLA 이론적 배경은 [19장](/wiki/chapters/chapter-19/)을 참조하세요.

**【왜】** VLA를 클라우드가 아닌 온보드에 배치해야 하는 이유는 세 가지 하드 제약 때문입니다: **지연 시간**(동작 명령은 네트워크 왕복을 견딜 수 없음), **연결성**(로봇이 Wi-Fi 없이 벽돌이 되어서는 안 됨), **프라이버시**(가정 환경 이미지가 외부로 유출되어서는 안 됨)(출처: [엣지 VLA 추론](/entry/ent_tech_on_device_vla_inference/) 카드). 그러나 엣지 배포의 대가는 모델이 제한된 VRAM 및 전력 예산에 맞춰져야 하며, 양자화, 가지치기, 증류 등의 압축 기술이 필요하다는 것입니다(출처: [안전 MCU](/entry/ent_component_safety_mcu/) 카드 트렌드 섹션). 기존 실증적 기준점: 300M 파라미터 확산 정책이 Orin NX 16GB에서 약 100ms 지연 시간을 보임(출처: data/roadmap/research/toddlerbot.md) – 100ms는 조작 작업에 사용 가능하지만, 동적 균형 유지를 위해서는 L1 계층의 소형 모델에 반드시 맡겨야 합니다.

**【당신의 상황 분석 방법】** 세 가지 질문에 솔직하게 답하세요: VLA 모델의 크기는 얼마인가(1B 미만이면 Orin NX/AGX Orin 고려 가능, 더 크면 Thor급을 보거나 클라우드 하이브리드 아키텍처 수용)? 동작 출력 빈도 요구사항은 얼마인가(조작 작업은 5–10Hz면 충분, 전신 동적 제어는 VLA 직접 출력에 의존할 수 없음)? 네트워크가 끊겼을 때 로봇이 반드시 유지해야 하는 능력은 무엇인가(최소한 온보드 안전 소형 정책 필요)? VLA 추론 성능은 모델 및 양자화 구성에 강하게 의존하므로, 구체적인 지연 시간 데이터는 VLA-Perf 등의 벤치마크 논문(출처: [엣지 VLA 추론](/entry/ent_tech_on_device_vla_inference/) 카드)을 참조하고 대상 하드웨어에서 실제 측정하세요.

## 검수 기준

- 작업 계층화 테이블 보유: 각 계산 작업에 목표 빈도, 허용 가능 지연 시간, 실패 결과가 표시되어 있으며, L0 작업에는 모두 독립적인 백업 경로(안전 MCU 또는 드라이버 로컬 루프)가 있습니다.
- 메인 컨트롤러 선정에 문서화된 이유 존재: "왜 이 보드인지" 설명할 수 있고, 최소 하나의 동급 오픈소스 프로젝트 구성을 대조로 제시(예: "RL 보행 + 소형 정책 수행, N95급 선택, BHL 참조").
- 실시간 Linux 경로 선택 시: 대상 하드웨어에서 `cyclictest`를 실행하여 무부하 및 부하 상태에서의 최대 스케줄링 지연 시간을 기록하고, 제어 루프 주기와 최소 한 자릿수 이상의 여유를 확보.
- VLA 계획 시: 대상 모델의 파라미터 수, 양자화 방식, 예상 VRAM 사용량을 기록하고, 선택한 플랫폼의 VRAM 용량이 30% 이상의 여유를 갖는지 확인.
- 전원 및 방열 계산 완료: 메인 컨트롤러 최대 전력 소비(예: AGX Orin 60W, Thor 75–120W급)가 전체 기기 전력 예산 및 방열 설계에 포함됨.

## 일반적인 문제 및 해결

| 증상 | 가능한 원인 | 해결 조치 |
|---|---|---|
| 로봇이 시뮬레이션에서는 안정적이나 실제 기기에서 떨림 발생 | 제어 루프 주기 미달, 스케줄링 지연 시간 변동 | `cyclictest`로 실제 최대 지연 시간 측정; PREEMPT_RT 패치 적용 여부, CPU 격리 및 메모리 잠금 확인 |
| 추론 실행 후 몇 분 만에 성능 급감 | 과열로 인한 클럭 저하(ToddlerBot 실제 측정 19분 후 클럭 저하, 출처: 해당 조사 파일) | 칩 온도 및 주파수 곡선 모니터링; 공기 흐름 개선/히트 스프레더 추가; 전력 모드 낮춰 재교정 |
| EtherCAT 간헐적 프레임 손실, 관절 오류 | 마스터 스케줄링 타임아웃 또는 DC 동기화 미설정 | 작업 카운터(Working Counter) 반환값 확인; 분산 클럭 활성화 확인; PREEMPT_RT + 전용 CPU 코어로 마스터 실행 |
| VLA 모델이 VRAM에 맞지 않거나 추론이 매우 느림 | 모델 과대, 미양자화 | 먼저 INT8/FP16 양자화 후 테스트; VRAM 요구사항과 플랫폼 사양 비교; 동작 청킹(action chunking)으로 추론 빈도 낮춤 |
| NUC/미니 PC에서 ROS 노드 간헐적 지연 스파이크 발생 | 표준 커널 스케줄링 지터 + USB 장치 인터럽트 경합 | 실시간 패치 적용; 인터럽트 친화성(IRQ affinity)을 비실시간 코어에 바인딩; CAN/IMU USB 어댑터를 독립 USB 컨트롤러로 교체 |
| 메인 컨트롤러 다운 후 로봇 제어 불능으로 넘어짐 | 독립 안전 계층 부재 | 안전 MCU 워치독 및 비상 정지 회로 추가하여 메인 컨트롤러 전원 차단 시 드라이버 자동 비활성화 구현 |
