Stage 0 基础筑基：拿到造人形机器人的入场券

⚠️ 本路线图内容基于公开资料与理论知识整理，未经实机验证；涉及电气与机械操作时请自行核实安全规范。

本阶段定位

Stage 0 的目标不是"学完所有基础"——那是个无底洞，无数人死在里面。目标是拿到四张入场券：

  1. 数学：看到控制论文里的矩阵、概率、动力学方程、QP 不陌生；
  2. 编程与工具链：Python/C++/git/ROS 2 能干活，知道 3D 打印是怎么回事；
  3. 电路概念：知道电从哪来、信号怎么跑，后面上真硬件不烧板；

  4. 仿真：读懂一份 URDF，并在 MuJoCo 里让双足模型站起来。

  5. 时间预算：业余 6–10 周（每周 8–10 小时），脱产 3–4 周。

  6. 预算：约 ¥0——本阶段软件全部开源；可选的 3D 打印打样体验费用需自行向供应商确认。
  7. 核心纪律：每个主题都给出"学到什么程度算够"的判据，达标即走，不许回头刷题。真正的缺口后面阶段会暴露，届时带着具体问题回来补，效率高十倍。

步骤 1：数学四件套

别按数学系的学法（定义-定理-证明）学，按"控制工程师最低弹药库"学。四门的判据都是可演示的，不是"我感觉懂了"。

1.1 线性代数（Linear Algebra）

【做什么】向量与矩阵运算、特征值/特征向量、旋转矩阵与齐次变换、最小二乘与伪逆。动手练习：用 NumPy 写 4×4 齐次变换，把足端坐标从踝关节系变换到躯干系；再写两连杆平面臂的正运动学。

【为什么】线性代数是人形机器人一切空间关系的语言：正运动学是齐次变换的连乘，雅可比矩阵把关节速度映射到末端速度，全身控制里的每个 QP 都是矩阵运算。卡片：线性代数（研究向量空间、线性变换、矩阵与线性方程组的数学分支）。Wiki 第 8 章的运动学推导全建在这上面。

【你的情况怎么分析】CS 背景：计算你早会了，重点补几何直觉——旋转矩阵不是九个数，是一个坐标系的三根轴；物理/数学背景：直接做上面的练习当验收；零基础：先看 3Blue1Brown《线性代数的本质》建立直觉，再上手算。

学到什么程度算够（三条全过即停）：① 能手推 2D 旋转矩阵并说清每一列的几何含义；② 两连杆平面臂正运动学代码能跑、输出和手算一致；③ 看到雅可比的伪逆 J⁺，知道它在解一个最小二乘问题。

1.2 概率论（Probability Theory）

【做什么】随机变量、期望与协方差、高斯分布、贝叶斯公式、条件化与边缘化。动手练习：用 NumPy 模拟带高斯噪声的"关节编码器读数"，画直方图、算均值/方差；再用贝叶斯公式手算一道"传感器报警→真故障概率"的题。

【为什么】真实传感器永远带噪声，状态估计（卡尔曼滤波一族）本质是递推的贝叶斯推断；后续用到的模仿学习、VLA 模型，输出的也是动作的概率分布而非确定值。卡片：概率论——机器人和机器学习中所有概率模型的根基。

【你的情况怎么分析】AI 背景：用判据自检，大概率直接过；硬件背景：这往往是你四门里最弱的一门，而它直接决定后面能不能看懂状态估计——值得把数学预算的一半投在这里；零基础：放到线性代数之后学。

学到什么程度算够：① 一句话说清协方差矩阵对角线与非对角线元素各是什么；② 贝叶斯题能手算且数值正确；③ 看到卡尔曼滤波的五个公式不慌——不要求会推导，要求说得出每个公式的角色（预测、增益、更新）。

1.3 经典力学（Classical Mechanics）

【做什么】牛顿-欧拉方程、拉格朗日方法、动能/势能、惯量张量、角动量。动手练习：用拉格朗日法写单摆、双摆的动力学方程，数值积分仿真 5 秒并画出角度曲线。

【为什么】人形机器人是一个浮动基多刚体系统，M(q)q̈ + C(q,q̇) + g(q) = τ 这个方程会贯穿你之后读到的每一篇控制论文；双足平衡（ZMP、捕获点）全是经典力学的直接推论。卡片：经典力学（含牛顿定律、守恒原理与刚体动力学）。Wiki 第 8 章与第 14 章 机器人控制基础的推导都从这里出发。

【你的情况怎么分析】机械/车辆背景：这门大概率够，做双摆练习验证即走；纯软件背景：这是你本阶段最重要的补课——没有力学直觉，仿真里机器人摔倒你都分不清是控制器的错还是模型的错。

学到什么程度算够：① 双摆方程能推（允许对着书推，但合上书能讲清每一步在干什么）；② 能定性解释"为什么躯干惯量大的人形机器人反而容易站稳"；③ 看懂 URDF 里 <inertial> 标签每个字段的含义——这直接为步骤 4 铺路。

1.4 凸优化（Convex Optimization）

【做什么】凸集/凸函数的判别、标准 QP（二次规划）形式、等式/不等式约束的几何意义、调用现成求解器解一个 QP。动手练习：用 Python 解 min ‖x‖² s.t. Ax=b, x≥0（求解器可选 OSQP、quadprog 等，按你的环境自行选择）。

【为什么】人形控制的实时层几乎都在解 QP：全身控制（WBC）把"任务跟踪 + 接触力约束 + 力矩上限"写成一个逆动力学 QP，每毫秒解一次；MPC 是滚动地解 QP。凸性保证任何局部最优即全局最优——这是工程师敢把 QP 放进 kHz 级控制环的根本原因。卡片：凸优化。深入见 Wiki 第 14 章。

【你的情况怎么分析】这是四件套里边际收益递减最快的一门：不自己写控制器的话，"会判凸 + 会调求解器 + 读懂 QP 结构"就够，KKT 推导留到以后按需补。运筹背景自检即过。

学到什么程度算够：① 给一个问题能判断是不是凸的（二次正定目标 + 线性约束 → 是）；② 练习题的解经检验满足全部约束；③ 能向别人解释"为什么 WBC 敢在实时环里用 QP"。

步骤 2：编程与工具链

【做什么】① Python 达到熟练：NumPy 向量化、matplotlib 画图、会写 class；② C++ 达到"能读能改"：指针/引用、头文件、编译链接；③ git 日常操作：clone、branch、commit、push、提 PR；④ 安装 ROS 2 LTS，跑通 talker/listener 并自写一对发布/订阅节点；⑤ 了解 FDM 3D 打印的基本流程（建模→切片→打印）与强度常识。

【为什么】人形软件栈的事实标准是 ROS 2 中间件——基于 DDS 的发布/订阅机制、支持实时（来源：ros.org，见卡片），后续阶段你的仿真、驱动、状态估计都以节点/topic 组织。底层实时环用 C++、算法原型用 Python，是行业默认分工。3D 打印则是后面整机阶段的制造主力：开源人形 Berkeley Humanoid Lite 的结构件与摆线减速器均为打印件，整机 BOM 美国采购约 $4,312、中国约 $3,236（来源：data/roadmap/research/berkeley-humanoid-lite.md）——你现在学的切片与公差常识，届时直接变成钱和时间。软件生态全景见 Wiki 第 22 章 软件中间件。

【你的情况怎么分析】纯软件背景：Python/git 自检通过就跳，时间投给 C++ 内存与实时概念、ROS 2 的节点/topic/service 三件套；硬件背景：从 NumPy 入门 Python 最划算，离 MATLAB 思维最近；零基础：先 Python 后 git，C++ 暂缓到需要时再学，学了不用会忘。

学到什么程度算够：① ROS 2：能独立写出两个节点一发一收，并用 ros2 topic echo 验证数据流动；② git：独立完成 fork→branch→commit→PR 全流程一次（拿自己的练习仓库练）；③ 3D 打印：能说出 FDM 层纹方向对零件强度的影响，以及为什么配合孔要留公差（具体机台的公差量级需自行向供应商确认）。

步骤 3：电路与嵌入式基础

【做什么】本阶段不画板子，只建立三组概念：供电（电压轨、电流预算、线径与压降、保险与开关）、地（共地、地环路）、信号（数字电平、差分传输、抗干扰）。然后重点理解 CAN 总线 的角色：它是连接人形机器人关节电机驱动器与中央控制器的现场总线（来源：卡片，Wikipedia CAN bus 2024）。可选动手项：买一块入门 CAN 适配器（价格需自行向供应商确认），用 can-utils 抓一段真实总线数据看看帧结构。

【为什么】到了整机阶段，你面对的将是十几到二十几个关节驱动器挂在总线上：Berkeley Humanoid Lite 就是用 CAN 连接各执行器，主控是一台 Intel N95 迷你 PC（约 $129，置于躯干，同时跑底层控制与 RL 策略；来源：data/roadmap/research/berkeley-humanoid-lite.md）。而整机调试时一半以上的"灵异问题"出在电气层：压降、共地、终端电阻、线束。现在建立概念，后面少烧板。电源系统工程细节见 Wiki 第 6 章 计算、电源与热管理。

【你的情况怎么分析】EE/自动化背景：本节 30 分钟过完概念即可；软件与零基础：不要求会设计电路，但必须做到"敢量、会量"——会用万用表量电压、量通断，知道 24 V 母线短路意味着什么。安全提示：锂电池与母线电容不是玩具，任何带电操作前自行核实安全规范，没把握就请有硬件经验的人在场。

学到什么程度算够：① 能徒手画出"电池→保险/急停→母线→各驱动器"的供电拓扑草图；② 能解释 CAN 总线两端为什么要加 120 Ω 终端电阻；③ 能解释差分信号为什么比单端信号抗干扰。

步骤 4：仿真初体验——读懂 URDF，让双足站起来

【做什么】① 安装 MuJoCo 物理引擎；② 找一份开源人形/双足的 URDF 机器人描述格式 文件通读——URDF 是用 XML 描述机器人连杆、关节、惯性与几何的格式（来源：wiki.ros.org/urdf，见卡片）；③ 把模型加载进 MuJoCo（或用项目自带的 MJCF）；④ 写一个最朴素的位置 PD 控制器：给定站立姿态的关节目标角，让双足模型原地站住 10 秒。

【为什么】MuJoCo 是具有丰富接触动力学的高保真物理引擎，广泛用于人形控制研究（来源：mujoco.org，见卡片）——双足站立的核心难点恰是脚底接触与平衡，正对它的强项。读懂 URDF 则是打开任何开源人形仓库的第一把钥匙：真实开源项目会同时维护 URDF/MJCF/USD 三种描述格式（Berkeley Humanoid Lite，来源：data/roadmap/research/berkeley-humanoid-lite.md），看懂这个格式，你就读得懂别人机器人的"骨架图纸"。仿真技术栈的系统论述见 Wiki 第 23 章 仿真与物理引擎。

【你的情况怎么分析】没有 GPU 不影响——站立级刚体仿真 CPU 足够。选模型优先"小尺寸、文档全"的开源双足（调研档案 data/roadmap/research/ 里有 Berkeley Humanoid Lite、ToddlerBot 等候选，附成本与门槛分析），别一上来就啃全尺寸机型。本阶段只能完成一件事的话，就完成这件——它同时验收力学、编程和工具链三件事。

学到什么程度算够：即下方验收标准第 1、2 条——本步骤就是 Stage 0 的毕业设计。

验收标准

逐项自检，全部通过才可进入 Stage 1：

  1. URDF 阅读理解：拿到一份没见过的双足 URDF，能不看文档口头回答——共几个可控关节、根连杆在哪、踝关节的 origin/axis 各是什么、<inertial> 中 ixx/iyy/izz 的含义；并用 check_urdf 或等价工具确认无解析错误。
  2. MuJoCo 站立：加载双足模型，用你自己写的位置 PD 控制实现无外力站立 ≥ 10 秒不倒；对躯干施加一个小冲量后，模型能恢复站立——或者你能正确解释它为什么不能恢复（增益、接触还是模型问题）。这条考的是力学直觉，不是调参运气。
  3. 数学抽查：四件套的"够的判据"里每条随机抽一款，能当场演示（手算或跑代码）。
  4. 工具链：ROS 2 talker/listener 有跑通记录（终端截图或录屏）；本阶段所有练习代码在你的 git 仓库里，提交历史清晰可读。
  5. 电路概念：能向同伴（或对空气）讲清供电拓扑、CAN 终端电阻与差分信号三件事；讲不清楚一律视为未达标。

两个"不算够"的警示信号：脱离教程写不出 PD 控制器（步骤 4 是跟出来的）；改了 URDF 的 origin 不知模型为什么飘（1.1 几何直觉没建立）。出现任一，回对应小节重验。

常见坑与排查

症状 可能原因 排查动作
MuJoCo 里模型一加载就散架/乱飞 初始姿态穿模（关节互相嵌入）；某 link 缺 <inertial> 导致质量/惯量为零 先看仿真器编译警告；给模型一个合理初始 keyframe；逐个 link 核对惯性参数
PD 站立时关节高频抖动 增益过高、积分步长过大，数值刚性 先把 P 减半再缓慢加回；减小仿真 timestep 复现对比；检查是否把期望阻尼写成了弹簧刚度
脚底打滑像溜冰 接触摩擦参数用了默认值，低于真实鞋底-地面组合 查 geom/接触对的 friction 参数；按鞋底材料（橡胶/打印 PLA）查摩擦系数量级调整，具体取值需按材料自行确认
URDF 转 MJCF 后关节运动方向反了 URDF axis 定义与目标仿真器约定不一致；角度单位混用（度 vs 弧度） 在 viewer 里单关节逐个驱动核对方向；全文检索角度字段确认单位
ROS 2 两节点互相收不到消息 ROS_DOMAIN_ID 不一致；防火墙拦截 DDS 流量；topic 名/QoS 不匹配 先在同机跑通再跨机；ros2 topic list 确认双方看到同一 topic；检查环境变量（以所用发行版官方文档为准）
迟迟不敢进 Stage 1，反复重学数学 完美主义陷阱：把"打基础"当成拖延借口 对照验收标准逐条打勾，达标即走；真正的缺口让 Stage 1 暴露

至此，Stage 0 完成。下一站 Stage 1 造一个关节（执行器）：把步骤 3 的电路概念和 PD 控制第一次用在真实硬件上；URDF 与 MuJoCo 的底子，则到 Stage 2 双足平台 的仿真行走里兑现。
