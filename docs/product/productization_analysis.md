# 产品化分析：从 AI4Sci 知识图谱工具到通用知识操作系统

> 角色：产品经理（PM）视角  
> 日期：2026-06-25  
> 前提：当前项目以「人形机器人完整产业链」为首个垂直领域，但底层模式（自然语言 → AI 工作流 → 知识图谱 → Web 可视化 + 问答）可复用于任意知识体系。

---

## 1. 一句话产品定位

**「AI-Native 知识操作系统」**：让用户通过自然语言，把任意领域的零散信息自动建构成一张可探索、可对话、可审阅的 living knowledge graph。

> 人形机器人知识图谱不是终点，而是第一个旗舰样例（flagship domain）。真正的产品是 underlying engine + 可替换的领域工作流。

---

## 2. 目标用户分层

虽然你说「用户是所有人」，但产品化必须选择一个**滩头用户（beachhead persona）**先跑通，再横向扩展。

| 层级 | 用户 | 画像 | 付费意愿 | 优先级 |
|------|------|------|----------|--------|
| **核心用户** | 技术型研究者 / 自学者 | 研究生、工程师、行业分析师、技术投资人 | 高（时间就是金钱） | P0 |
| **付费客户** | 研发团队 / 咨询公司 | 机器人公司、VC、智库、高校课题组 | 极高（团队席位） | P1 |
| **扩展用户** | 通识学习者 | 对某个领域好奇的大众 | 中低 | P2 |
| **贡献者** | 领域专家 / KOL | 愿意审阅和补充内容以建立声誉 | 非直接付费，但降低内容成本 | P1 |

**建议滩头用户**：
> **"需要快速搞懂一个复杂技术领域，并做出有来源支撑决策的人"** —— 例如机器人公司战略部、投资机器人赛道的分析师、研究生开题前的文献调研。

## 3. 核心场景（结合你的描述定义）

### 场景 A：探索一张已经存在的知识图谱
> **用户说**："谐波减速器和行星减速器有什么区别？"

**产品响应**：
1. 在图谱中高亮两个节点及其共同邻居。
2. 侧边栏显示对比摘要、来源、关键关系。
3. 底部 chat 给出带引用 ID 的中文回答。

### 场景 B：让 AI 自动填补知识空白
> **用户说**："帮我调研一下人形机器人目前主流的执行器方案。"

**产品响应**：
1. 系统检查当前图谱缺口。
2. 自动触发 `actuator_module_design` 等工作流。
3. 生成 entries + relationships，进入 staging。
4. 审阅后进入 production 图谱，用户收到通知。

### 场景 C：结构化学习一个领域
> **用户说**："我想从零开始学习人形机器人。"

**产品响应**：
1. 基于图谱拓扑生成学习路径（Foundations → Components → AI → Integration）。
2. 每个节点提供「阅读 → 问答 → 小测验」闭环。
3. 学习进度可视化。

### 场景 D：审阅 AI 生成的内容
> **用户说**："这批新抓取的论文质量如何？"

**产品响应**：
1. 打开 Review Inbox，列出所有 staging drafts。
2. 每页显示原文摘要、AI 提取、来源链接、置信度。
3. 一键 Approve / Reject / Edit，支持批量操作。

### 场景 E：监控领域动态
> **用户说**："这周人形机器人领域有什么新进展？"

**产品响应**：
1. 后台定时运行工作流扫描 arXiv / 网络。
2. 生成「本周新增/变更」feed。
3. 用户可订阅关键词或子图谱。

## 5. 交互优先级

你提出「图谱可视化优先，在图谱上有 chat 输入框」，这个判断非常正确。以下是建议的 UX 层级：

1. **P0：图谱画布（Graph Canvas）**
   - 首页就是图谱，这是产品 identity。
   - 支持缩放、搜索节点、点击展开关系、聚焦子图。
   - 不同颜色/形状区分 entity type 和 layer。

2. **P0：嵌入式 Chat 输入框**
   - 固定在图谱底部或右下角。
   - 用户可输入问题，也可选中节点后问「这个和 X 有什么关系？」。
   - 回答中引用的节点可点击并高亮。

3. **P1：侧边详情卡（Entity Card）**
   - 点击节点后右侧滑出：名称、摘要、领域、关系、来源、操作按钮。
   - 操作包括「追问」「添加到学习路径」「发起审阅」。

4. **P1：Review 面板**
   - 对 contributor / expert 用户显示 pending drafts。
   - 对普通用户隐藏或只读显示「本条目由 AI 生成，尚未人工审阅」标记。

5. **P2：工作流触发器**
   - 在 chat 里用自然语言触发："帮我补一下电池领域的知识"。
   - 也可提供显式按钮："Discover new papers"。

## 6. 工作流产生的内容与审阅机制设计

这是产品可持续性的核心。建议采用 **"AI 生成 + 人工审阅 + 社区反馈"** 三层机制。

### 6.1 内容生成工作流

每个 workstream 是一个可配置的工作流模板：

```yaml
workstream: humanoid_actuators
sources: [arxiv, semantic_scholar, web]
prompts:
  discovery: "Find papers about..."
  extraction: "Extract entities and relationships..."
  synthesis: "Write multi-lingual summaries..."
schema_constraints:
  allowed_types: [paper, component, method]
  required_fields: [sources, verification]
```

触发方式：
- **用户主动**：chat 中表达需求 → 系统匹配 workstream。
- **定时任务**：daemon 按 schedule 扫描。
- **缺口驱动**：系统发现某领域节点稀疏，自动建议补充。

### 6.2 审阅机制

建议采用 **"分级审阅"**，而非所有内容都人工看：

| 内容类型 | 审阅方式 | 说明 |
|----------|----------|------|
| 新增 entry | 抽样 + 置信度阈值 | 高置信度自动过，低置信度强制审 |
| 新增 relationship | 批量审阅 | 一个 workstream 的关系集中审 |
| 编辑已有内容 | 差异审阅（diff） | 只看改了什么 |
| 冲突声明 | 强制人工仲裁 | 多个来源结论矛盾时 |

**审阅界面要点**：
- 左侧：AI 生成的 entry / relationship。
- 右侧：原始来源摘要、置信度、schema 校验结果、dangling 警告。
- 操作：Approve / Reject / Edit / Request Evidence / Skip。
- 状态标签：AI-proposed → Under Review → Verified → Rejected。

### 6.3 质量信号

为了让用户信任内容，产品需要显式展示：

- **Verification badge**：未审阅 / AI 提取 / 人工核验 / 多方验证。
- **Source strength**：arXiv 论文、公司财报、官方文档、博客（权重不同）。
- **Reviewer reputation**：审阅者的历史准确率。
- **Conflict flag**：当两个来源结论矛盾时提示用户。

## 7. 订阅付费模型

按你的方向，建议 **SaaS 订阅制**，按用户深度和功能权限分层：

| 套餐 | 月费（参考） | 目标用户 | 核心权益 |
|------|-------------|----------|----------|
| **Free** | $0 | 大众探索者 | 公开领域图谱浏览、每日 10 次 chat、基础搜索 |
| **Learner** | $9 | 自学者 / 学生 | 无限 chat、学习路径、导出笔记、无广告 |
| **Pro** | $29 | 研究员 / 分析师 | API 访问、高级查询、自定义监控、私有 workspace |
| **Team** | $99/席位 | 企业研发团队 | 私有领域、自定义 workstream、SSO、审阅席位、审计日志 |

**计费指标建议**：
- 主要：订阅档位（功能+席位）。
- 辅助：超出套餐的 AI 工作流执行次数、API 调用量、私有存储空间。

> 注意：不要把「按 token 计费」作为主力，用户讨厌不可预测的成本。订阅制更利于商业化。

## 8. 90 天 Roadmap（产品化方向）

| 阶段 | 时间 | 目标 | 关键交付 |
|------|------|------|----------|
| **Month 1** | 0–30 天 | 让现有机器人图谱可被消费 | 稳定 Web 前端、Graph + Chat 一体化、基础搜索/问答 |
| **Month 2** | 31–60 天 | 建立人-in-loop 审阅闭环 | Web Review Inbox、Approve/Reject/Edit、verification badge |
| **Month 3** | 61–90 天 | 验证通用性与付费意愿 | 抽象 workstream 配置、支持第二个 demo 领域、上线订阅页 |

## 9. 风险与差异化

### 风险
1. **内容质量失控**：AI 生成大量低质量条目会稀释图谱价值。
   - 对策：强制审阅 + 质量信号 + 逐步放开自动化程度。
2. **通用平台陷阱**：如果追求「任何领域都能用」，可能哪个领域都做不深。
   - 对策：先在一个垂直领域（机器人）做到足够好，再横向复制。
3. **学术版权与来源风险**：抓取论文、网页内容需注意版权和引用规范。
   - 对策：始终保留原始来源链接，不存储全文，只存储摘要和提取的实体关系。

### 差异化
- 不是「ChatGPT + 搜索」，而是**图谱是核心资产**。
- 不是「静态百科」，而是**AI 持续自动补全 + 人工审阅**的 living graph。
- 不是「单一领域工具」，而是**可配置的领域工作流引擎**。

---

## 10. 给下一步的建议

1. **把机器人图谱做成一个令人惊叹的 demo**：让任何人打开 Web 都能直观感受到「这就是未来查知识的方式」。
2. **优先做 Web 审阅面板**：让非工程师也能参与内容生产，这是产品化的关键。
3. **开始抽象 workstream SDK**：把 `scripts/ai4sci_workstreams/` 变成可配置、可分享的模板。
4. **选一个第二领域做验证**：例如「固态电池」或「大模型推理优化」，验证通用性。
5. **尽早设计订阅页**：哪怕还没上线，也要有清晰的付费边界，指导功能优先级。

---

**结论**：这个项目确实非常有前景。它本质上是在做**「知识的自动化生产与消费基础设施」**。人形机器人是一个很好的起点，但产品化的真正机会在于把这套模式变成可复用的平台。
